CHAPTER 6
Technology Comparison and Technology Used
Project: Development of an Application for AI-Enhanced Game-Based Learning
This chapter compares the main technologies considered for the proposed platform and justifies the final stack selected for the MVP. The discussion is organized around the project's practical requirements: a mobile-first user experience, low-latency quiz submission, controlled AI assistance, maintainable service boundaries, and implementation feasibility within an academic project scope.
6.1 Technology Selection Criteria
The technology stack was not chosen only on the basis of popularity. Instead, it was selected according to the technical and educational characteristics of the project. The proposed application must support a mobile client, structured learning content, quiz attempt processing, gamification updates, teacher analytics, and AI-assisted feedback. In addition, the platform has to remain secure, auditable, and realistic to implement within one academic term. Therefore, the comparison focused on six practical criteria: performance, scalability, ease of development, maintainability, integration capability, and security/privacy support.
Table 6.1. Technology Selection Criteria
CriterionImportance for the ProjectPerformanceThe system must keep quiz submission fast and predictable, while AI help should remain responsive or degrade safely under external model latency.ScalabilityAI assistance and analytics workloads should scale independently from the critical quiz submission flow.Ease of developmentThe stack should allow rapid MVP implementation by a small student team and support short iteration cycles.MaintainabilityThe project should support future changes in scoring logic, gamification rules, and AI providers without major rewrites.Integration capabilityChosen tools should integrate well across mobile UI, backend APIs, authentication, storage, and external AI services.Security and privacyBecause the platform may process student data and AI interactions, the stack must support authentication, access control, auditability, and controlled external communication.6.2 Comparison of Main Technologies
The project presentation defines the planned stack as Flutter for the frontend, Python with FastAPI for the backend, Google Firebase for database and authentication, LLM API with n8n for the AI assistant, S3/MinIO for learning materials, and Git, Docker, pytest, and Flutter tests for engineering support. The architecture package further clarifies why this stack fits a mobile-first, service-oriented design with a controlled AI gateway and asynchronous downstream processing. For the client layer, the most relevant alternatives were Flutter, React Native, and native Android development. Native Android can offer strong device-level control and platform-specific optimization, but it would narrow the MVP to one operating system and increase implementation effort. React Native is also a cross-platform option, but it depends more heavily on the JavaScript ecosystem and typically requires additional integration effort for highly consistent UI behavior across screens.
Flutter was selected because it fits the mobile-first nature of the project while allowing the team to build one coherent interface in a single codebase. This is especially useful for a gamified educational platform, where visual consistency, smooth navigation, animated progress indicators, and reusable UI components are important. Flutter also matches the project plan, where frontend work focuses on core screens, user flows, gamification UI, and the AI assistant interface. In practical terms, Flutter offers a good balance between development speed and interface quality, making it more suitable than fully native development for a semester-bounded MVP.
6.2.2 Backend and API Layer: Python with FastAPI
At the backend level, the main alternatives were FastAPI, Django REST Framework, and a Node.js-based API framework such as Express or NestJS. Django REST Framework is mature and feature-rich, but it is comparatively heavier for a project whose main need is a clean API layer and orchestration of separate services. A Node.js stack is also viable, especially for event-driven systems, but it would split the technology focus between Python-oriented AI work and JavaScript-oriented backend work.
FastAPI was chosen because it is lightweight, well-suited to REST API development, and fits naturally with Python-based AI integration. In this project, the API Gateway/BFF is responsible for request validation, orchestration of internal service calls, and versioned client-facing APIs. FastAPI supports this role well, while keeping the backend simple enough for rapid development and testing. Its compatibility with pytest further supports maintainability and automated validation. The selection therefore improves developer productivity and reduces friction between backend development and AI-related modules.
6.2.3 Database and Authentication: Firebase / Firestore / Firebase Auth
For persistent data and identity management, three broad options were considered: Firebase with Firestore and Firebase Authentication, a traditional relational database such as PostgreSQL with custom authentication, and a managed backend platform such as Supabase. A PostgreSQL-based design provides strong relational guarantees and advanced querying, but it also increases setup and operational complexity, especially when combined with custom authentication and role handling. Supabase is attractive as a managed platform, but in the current project materials Firebase is already treated as the trusted identity provider and primary database, which aligns more directly with the chosen architecture.
Firebase was selected because it simplifies MVP delivery while still supporting the project's main requirements. Firebase Authentication provides token-based identity flows, while Firestore can support structured educational data, progress records, and service-specific adapters. This choice also fits the architecture package, where IAM verifies Firebase tokens and where Firestore is accessed through repository adapters rather than being exposed directly to all services. The result is a stack that is easier to implement quickly than a fully self-managed relational setup, while still leaving room for later abstraction and migration if needed.
6.2.4 AI Integration and Orchestration: LLM API with n8n
The AI layer is one of the most distinctive parts of the project, so several integration patterns were possible. The simplest option would be to call an LLM API directly from the backend. Another option would be to adopt a heavier orchestration framework and embed all prompt routing and workflow logic directly in application code. The project instead selected a hybrid model: a dedicated AI Tutor Gateway in the backend, combined with n8n for workflow orchestration and an external LLM API for generation.
This choice is justified by both system quality requirements and educational safety needs. The architecture documents emphasize that only the AI Tutor Gateway should communicate with external AI systems, and that AI help must support safety filters, rate limits, caching, logging, and fallback behavior. n8n is useful in this context because it externalizes prompt flows and routing logic, making the AI pipeline easier to iterate on without tightly coupling every change to core backend code. The final arrangement - AI Tutor Gateway plus n8n plus LLM API - provides enough flexibility for hint generation, explanation flows, and adaptive feedback while keeping the rest of the system protected from direct model dependency.
6.2.5 File Storage: S3/MinIO
The platform needs separate handling for media assets and learning materials, such as uploaded files, images, or attachments. The main alternatives were S3/MinIO, Firebase Storage, and local server storage. Local storage is the least suitable option because it is not robust for deployment, scaling, or multi-service access patterns. Firebase Storage would integrate well with the Firebase ecosystem, but the architecture package already assumes storage abstraction through a dedicated content/storage adapter rather than tightly coupling all files to a single managed backend vendor.
S3/MinIO was selected because it fits that adapter-based architecture and cleanly separates structured data from file storage. In practice, metadata can remain in the content domain, while large files are stored in an object store designed for that purpose. This improves maintainability, reduces schema pollution in the main database, and offers flexibility between cloud and self-hosted deployment models. For an academic MVP, this is a pragmatic middle path: simple enough to use, but architecturally cleaner than keeping all binary assets inside the core database layer.
6.2.6 Testing, Version Control, and Deployment Support
The support toolchain is also part of the technology decision. The project presentation identifies Git and Docker as core engineering tools, with pytest for backend testing and Flutter tests for UI and integration checks. Alternative approaches such as purely manual testing or unmanaged environment setup would be faster in the short term, but they would reduce reproducibility and make integration problems harder to detect.
The selected toolchain strengthens the overall quality of the project. Git supports collaboration, rollback, and branch-based feature development. Docker improves environment consistency across developer machines and future deployment targets. pytest is appropriate for validating backend APIs, scoring logic, and internal service behavior, while Flutter tests are suitable for checking UI flows and client-side correctness. Together, these tools increase reliability without adding disproportionate complexity, which makes them highly appropriate for a student team delivering a working MVP.
6.3 Rationale for the Final Technology Stack
The final technology stack is coherent because each selected tool supports a specific architectural responsibility instead of duplicating the role of another layer. Flutter addresses the mobile user experience; FastAPI supports the public API and service orchestration; Firebase handles authentication and structured cloud data; S3/MinIO stores larger learning materials; n8n and the LLM API enable flexible AI workflows; and Docker, Git, pytest, and Flutter tests support reproducible development and validation. This separation of concerns is important for a project that combines standard learning workflows with AI-enhanced behavior.
Another reason for these choices is that they align with the project architecture. The Mobile App communicates only with the API Gateway/BFF. Quiz submission remains a protected core flow, while gamification and analytics are updated asynchronously. The AI Tutor is isolated behind a dedicated gateway rather than being embedded across the whole backend. Such design decisions improve maintainability and allow future modifications - for example, replacing a database adapter or changing the external model provider - without forcing a full redesign of the system.
From a practical perspective, the selected technologies also match the team's work allocation shown in the project plan. Frontend work is centered on Flutter screens and the gamification UI, backend work is centered on FastAPI, authentication, and logic, and AI work is centered on prompt design, hint generation, adaptive feedback, and LLM integration. Because the technologies map clearly to team responsibilities, the stack is not only technically appropriate but also organizationally realistic for MVP delivery.

In conclusion, the chosen technologies are suitable for the development of an AI-enhanced game-based learning application because they support the project's educational goals and software quality requirements at the same time. Flutter enables a consistent and engaging mobile interface, FastAPI provides a clean and efficient backend layer, Firebase reduces authentication and database complexity for the MVP, n8n with an LLM API supports flexible AI tutoring workflows, and S3/MinIO keeps file storage separate from structured application data. The supporting toolchain of Git, Docker, pytest, and Flutter tests further strengthens implementation quality. As a result, the final stack is not only technically defensible, but also realistic, maintainable, and well aligned with the scope of the diploma project.
